Seven Principles of Object-Oriented Design

TechnologySoftware EngineeringSoftware Engineering设计模式

seven principles

Opening and closing principle, Liskov substitution principle, Demeter principle (least known principle), single responsibility principle, interface separation principle, dependency inversion principle, combination/aggregation reuse principle.

The Open-Closed Principle (OCP)

Detailed explanation of the opening and closing principle

https://omegaxyz.com/2018/06/14/Opening and closing principle of object-oriented design principles/
Software entities (modules, classes, methods, etc.) should be open for extension and closed for modification.

Conceptual understanding

The opening and closing principle means that in object-oriented design, when designing classes or other program units, you should follow:
  • open to extension
  • The design principle of being closed to modifications.
The opening and closing principle is one of the most basic principles for judging whether object-oriented design is correct.
According to the open-close principle, when designing a software system module (class, method), it should be possible to expand its functionality (open by extension) without modifying the original module (modify and close).
  • Extension and openness: The function of a module is extensible, then the module is open for extension. The functional scalability of a software system requires that the modules be open to expansion.
  • Modification closed: A module is called by other modules. If the source code of the module does not allow modification, the module modification is closed. The functional stability and continuity of the software system require that the module be modified and closed.

How to implement the open-close principle

In order to satisfy the open-closed principle, the closed-for-modification principle and the open-for-expansion principle, the unchanged parts of the software system should be abstracted. In object-oriented design,
  • These immutable parts can be abstracted into immutable interfaces, and these immutable interfaces can cope with future expansion;
  • Minimum functionality design principle for interfaces. According to this principle, the original interface can either cope with future expansion; the deficiencies can be achieved by defining new interfaces;
  • Calls between modules are made through abstract interfaces, so that even if the implementation layer changes, there is no need to modify the caller's code.
Interfaces can be reused, but interface implementations may not necessarily be reused.
The interface is stable and closed, but the implementation of the interface is changeable and open.
Flexible expansion of the software system can be achieved by adding new functions to the system or changing the original functions of the system through different implementations of interfaces and class inheritance behaviors.
benefit:Improve the reusability and maintainability of the system.
Simply put, whether a software system has a good interface (abstract) design is an important criterion for judging whether a software system satisfies the open-close principle. Nowadays, the open-close principle is often equated with interface-oriented software design.

The relativity of the open and closed principle

The construction of a software system is a process that requires constant reconstruction. In this process, the functional abstraction of modules and the relationship between modules will not be very clear from the beginning. Therefore, it is quite difficult to build a software system that 100% satisfies the open-close principle. This is the relativity of the open-close principle.
However, in the design process, through the abstraction of module functions (interface definition), the abstraction of the relationship between modules (through interface calls), the separation of abstraction and implementation (interface-oriented programming), etc., we can come as close as possible to satisfying the opening and closing principle.

Liskov Substitution Principle (LSP)

Only OO designs that meet the following two conditions can be considered to meet the LSP principle:
  • Conditions such as if/else that determine the type of derived classes should not appear in the code.
  • The derived class should be able to replace the base class and appear anywhere the base class can appear, or if we replace the base class in the code with its derived class, the code will still work normally.
The Liskov Substitution Principle (LSP) is an important guarantee to make the code comply with the opening and closing principle.
LSP embodiment
  • Class inheritance principle: If an object of a derived class may cause a runtime error where the base class appears, the derived class should not inherit from the base class, or in other words, shouldredesignrelationship between them.
  • Action correctness guarantee: On the other hand, it is guaranteed that extensions of classes that comply with LSP design principles will not introduce new errors into the existing system.

The extended significance of the Li-style substitution principle

Subclasses can extend the functions of the parent class, but cannot change the original functions of the parent class.
  • Subclasses can implement abstract methods of the parent class, but cannot override non-abstract methods of the parent class.
  • Subclasses can add their own unique methods.
  • When a method of a subclass overrides a method of a parent class, the method's preconditions (i.e., the input/parameters of the method) are looser than the input parameters of the parent class method.
  • When a method of a subclass implements a method of a parent class (overloading/overriding or implementing an abstract method), the postconditions (i.e. the output/return value of the method) are stricter or equal to those of the parent class.

Advantages of Li substitution principle

  • The proliferation of constraint inheritance is a manifestation of the open-close principle.
  • Strengthen the robustness of the program, and at the same time, it can also greatly improve the maintainability and scalability of the program when making changes. Reduce risks introduced when requirements change.

Law of Demeter (LoD)

Conceptual understanding

For OOD-oriented, it is interpreted in the following two ways:
1) A software entity should interact with other entities as little as possible.
2) Each software unit has minimal knowledge of other units and is limited to those software units that are closely related to its own unit.
Confirmation of circle of friends
"Friend" conditions:
  1. The current object itself (this)
  2. The object passed into the current object method as a parameter
  3. The object directly referenced by the instance variable of the current object
  4. If the instance variable of the current object is a collection, then the elements in the collection are also friends.
  5. The object created by the current object
Any object, if it meets one of the above conditions, is a "friend" of the current object, otherwise it is a "stranger".

Advantages and Disadvantages of Demeter's Principle

The original intention of the Dimit principle is to reduce the coupling between classes. Since each class minimizes its dependence on other classes, it is easy to make the functional modules of the system functionally independent, with no (or very few) dependencies between each other.
The Demeter principle does not want classes to establish direct contact directly. If there is really a need to establish contact, I hope it can be conveyed through its friend class. Therefore, one possible consequence of applying the Dimit Principle is that there are a large number of intermediary classes in the system. These classes exist solely to transfer the mutual calling relationships between classes, which increases the complexity of the system to a certain extent.

single responsibility principle

Never give a class more than one reason to change.
In other words, if a class needs to be changed, there is always only one reason to change it. If there are multiple reasons to change it, the class needs to be redesigned.
The core meaning of the single responsibility principle is: only one class/interface/method can have one and only one responsibility.
Why can't a class have more than one responsibility?
If a class has more than one responsibility, there will be many different reasons for the class to change, and this change will affect the users of the class with different responsibilities (different users):
  • On the one hand, if one responsibility uses an external class library, users using another responsibility must also include the unused external class library.
  • On the other hand, if a user needs to modify one of the responsibilities for some reason, the user of the other responsibility will also be affected and he will have to recompile and configure.
This violates the opening and closing principle of the design and is not what we expect.

Interface Segregation Principle (ISP)

Users cannot be forced to rely on interfaces they do not use.

Conceptual understanding

In other words, it is always better to use multiple specialized interfaces than a single overall interface.
It contains 2 levels of meaning:
  • Interface design principles: The design of interfaces should follow the minimum interface principle, and do not stuff methods that users do not use into the same interface. If the methods of an interface are not used, it means that the interface is too fat and should be divided into several interfaces with specialized functions.
  • Interface dependency (inheritance) principle: If an interface a inherits another interface b, then interface a is equivalent to inheriting the method of interface b. Then interface a after inheriting interface b should also follow the above principles: it should not contain methods that are not used by users. On the contrary, it means that interface a is polluted by b, and their relationship should be redesigned.
If users are forced to rely on an interface they don't use, they will have to change when the interface changes. In other words, if a user relies on an interface that is not used but is used by other users, when other users modify the interface, all users who rely on the interface will be affected. This obviously violates the open-closed principle and is not what we expect.

Guidance

  1. The dependence of a class on a class should be based on the smallest interface
  2. Build a single interface, don’t build a huge and bloated interface
  3. Try to refine the interface as much as possible and keep as few methods in the interface as possible

Dependency Inversion Principle (DIP)

  • High-level modules should not depend on low-level modules, both should depend on abstractions
  • Abstractions should not depend on details, details should depend on abstractions C. Program for interfaces, not implementations.

Conceptual understanding

rely:In programming, if a module a uses/calls another module b, we say that module a depends on module b.
High-level modules and low-level modules: Often in an application, we have some low-level classes that implement some basic or elementary operations, which we call low-level modules; in addition, there are some high-level classes that encapsulate some complex logic and depend on low-level classes. These classes are called high-level modules.
Dependency Inversion (Dependency Inversion）
In object-oriented programming relative to process-oriented (structured) programming, dependencies are inverted. Because in traditional structured programming, high-level modules always depend on low-level modules.
The abstract interface is an abstraction of the low-level module, and the low-level module inherits or implements the abstract interface.
In this way, high-level modules do not directly depend on low-level modules, but rely on the abstract interface layer. The abstract interface also does not depend on the implementation details of the low-level module, but the low-level module depends on (inherits or implements) the abstract interface.
Relationships are established between classes through the abstract interface layer.
Composite/Aggregate Reuse Principle (CARP)
Try to use composition/aggregation instead of class inheritance.

Conceptual understanding

That is, using some existing objects in a new object to make them part of the new object. The new object achieves the purpose of reusing existing functions by delegating to these objects. This means that composition and aggregation should be used as much as possible instead of inheritance relationships to achieve reuse.
Combinations and aggregations are special kinds of associations.
Aggregation represents the relationship between the whole and its parts and represents "possession". Combination is a stronger "possession", and the life cycle of the part and the whole is the same.
The combined new object has complete control over its component parts, including their creation and annihilation. Component objects of one composition relation cannot be shared with another composition relation.
Combination is the aggregation of values ​​(Aggregation by Value), and the general aggregation is the aggregation of references (Aggregation by Reference).
In object-oriented design, there are two basic ways to achieve reuse: the first is through composition/aggregation, and the second is through inheritance.
