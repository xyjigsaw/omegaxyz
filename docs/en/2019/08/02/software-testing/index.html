Software testing - test plan

1. Introduction

1.1 Overview

This test is conducted on the PC web version of the computer program ability online assessment system. The website of the assessment system ishttps://www.patest.cn/Users and administrators can enter the system according to corresponding permissions. This test plan will test the software under test from several aspects, including function decomposition, design test methods, and design test cases and steps. This document mainly explains some details of the online examination system testing process, and provides a framework and specifications for the online examination system testing work.

1.2 Online Assessment System Test Purpose and Test Tasks

In the process of developing this computer program ability online assessment system (PTA), many methods and technologies to ensure software quality were used (including permission management, test question distribution, and high-concurrency online assessment). However, many errors and defects will still be hidden in the developed software. This is especially true for this system. Therefore, strict software testing plays an important role in ensuring the quality of this online evaluation system. The purpose of software testing determines how to organize testing. If the purpose of testing is to find as many errors as possible, then the testing should be directed at the more complex parts of the software or the locations where there have been more errors in the past. The basic task of the testing phase should be to carefully design a set of "high-yield" test cases (a set of input data and corresponding expected output results) based on the documentation at each stage of software development and the internal structure of the program. When designing test cases, reasonable input data and unreasonable input data should be included. Use these use cases to execute the program and find potential defects in the software. A good test case is likely to find use cases that have not yet been discovered; a successful test refers to a test that reveals so far undiscovered defects. Subjectively due to the limitations of developers' thinking, and objectively due to the considerable complexity of currently developed software systems, software errors are inevitable during the development process. If errors in development can be eliminated early, troubles caused to later work can be eliminated, and high costs can be avoided, thereby greatly improving the efficiency of the system development process. Therefore, software testing is indispensable in all aspects of the entire software development life cycle. The main purpose of this test document: to determine the strategy, scope and methods of project testing; to enable all participants in the project testing work (test managers, testers) to have a clear understanding of the goals, scope, strategies, methods, organization, resources, etc. of the project testing work; to enable all participants in the project testing work to understand the test control process; to explain the organization and management of the project testing from a strategic perspective, guide the testing progress, and serve as the basis for the implementation of the project testing work;This document is the basis, specification and standard for the entire testing process of this project.

The intended readers of this Test Plan are:
  • Program teaching platform development manager
  • Technical manager
  • Test department manager
  • All members of the test team (includingSQA）

3.1 Test process

picture3.1 Test flow chart

3.2 Overview of test methods

The tests of this online assessment system include:
  • Functional testing tests whether each function is defective
  • Interface testing tests the performance data of the interface in a certain environment
  • Performance testing
  • stress test
  • Security testing
  • When testers perform tests, they must strictly follow the content in the test cases to perform the test work
  • Testers should record the test execution process in the test execution record document
  • Testers should record problems found during testing into defect records
  • test organization
 

3.3 Test the overall strategy

  • Guided by the Pareto principle. Try to find as many defects as possible (especially serious defects) in the limited time
  • Test planning is carried out simultaneously with requirement formulation and use case design
  • Testing requirements must be developed. By determining the content to be tested and their respective priorities and importance, the test design work can be made more purposeful, and more and more effective use cases can be designed under the guidance of requirements.
  • Gradually improve the test case library. The construction of the test case library is a process of continuous improvement. We must first design a complete set of test cases in a limited time. The important part of the test cases needs to be designed more fully, and the general parts need to point out the key points of the test. We will continue to improve the test case library in future testing work.
  • The testing process needs to be controlled. Carry out tests according to the pre-defined test execution sequence and fill in the test record form to ensure that the test process is under control.
  • Determine the focus. The focus of the test is on the functional implementation of each subsystem, with the provincial central management system and certificate management system, which have many problems, being the top priority.

3.5 Test risk analysis

  • Risks related to testers’ familiarity with the system: The testers participating in this project are testing the system for the first time and may not fully grasp the business details of system testing. This will cause some test escapes in subsequent test design and test execution work (that is, some aspects to be tested are not tested).
  • Risks in system data: The system being tested in this project does not have complete development documentation. Testers can only refer to personal experience when doing test design, which may result in testers being unable to conduct comprehensive and in-depth testing of the system in the early stages.
  • Time risk: This project is relatively short, but it has to complete the formulation of test specifications, the design of a complete set of test cases and the execution of a complete round of testing. The time schedule is very tight, which may lead to incomplete test design work.

3.10 Stress testing

Stress test requirements:
  • Determine the scale of inventory data (users generally require the production of 3-5 years of inventory data);
  • Determine the businesses that need to be stress tested (generally the businesses that are most frequently used by users or have complex business operations);
  • Determine the number of operating users and the proportion of various types of operating users;
  • Peak business volume requirements (generally the maximum number of transactions to be processed in one hour);
  • Requirements for real-time business response time (such as the processing time of a single business (such as less than 60 seconds) under peak conditions);
  • Time requirements for the batch processing process (such as the time for day-end (month-end, year-end) processing and batch data transmission with external systems).
  Stress test scenario:
  • Single user running stress test scenario;
  • Run test scenarios with multiple users, check system resource consumption and other tuning work begins.
  Stress testing tools:LoadRunner
