Programming implementation of combination mode and chain of responsibility mode

Programming LanguagesC & C++TechnologySoftware EngineeringC++Software Engineering设计模式

Combination mode

Introduction
  • Group objects into a tree structure to represent a part-whole hierarchy.
  • The composition pattern allows users to use single objects and composite objects consistently.
motivation
  • The headquarters, branches and offices are in a tree structure, that is, they have an organizational structure and cannot be simply managed in parallel.
  • It is hoped that the organizational structure of the head office, such as the management functions of the human resources department and finance department, can be reused in branch offices. This is actually a question where the whole and the parts can be treated consistently.
applicability
  • Want to represent a "part-whole" hierarchy of objects.
  • It is hoped that users will ignore the difference between composite objects and single objects, and users will use all objects in the composite structure uniformly.
structure participants
  • Component
    • Declare an interface for the objects in the composition.
    • Where appropriate, implement the default behavior of interfaces common to all classes.
    • Declare an interface for accessing and managing subcomponents of Component.
    • Define an interface in a recursive structure that accesses a parent widget and implements it where appropriate. (optional)
  • Leaf
    • Represents leaf node objects in a combination. Leaf nodes have no child nodes.
    • Defines the behavior of leaf node objects in a composition.
  • Composite
    • Defines the behavior of those components that have subcomponents.
    • Store subparts.
    • Implement operations related to subcomponents in the Component interface.
  • Client
    • Manipulate the object of the composite component through the Component interface.
  cooperation
  • Users use the Component class interface to interact with objects in the composite structure.
  • If the recipient is a leaf node, the request is processed directly.
  • If the recipient is a Composite, it usually sends the request to its subcomponents, possibly performing some auxiliary operations before and/or after forwarding the request.
Effect
  • Defines a class hierarchy containing basic objects and composite objects.
  • Simplify client code.
  • Makes it easier to add new types of components.
 

chain of responsibility model

Introduction
  • To decouple the sender and receiver of the request so that multiple objects have the opportunity to handle the request.
  • Connect these objects into a chain and pass the request along the chain until an object handles it.
motivation
  • Consider leave requests in your company.
  • The management personnel in the company are divided into levels from low to high: manager, director, and general manager.
  • For leave, you need to apply to managers at different levels according to the length of the day. For example, if it is less than 2 days, you can apply directly to the manager. If it is less than a week, it must be approved by the director. If it is longer, it needs the approval of the general manager.
applicability
  • Multiple objects can handle a request, and which object handles the request is determined automatically at runtime.
  • Want to submit a request to one of multiple objects without explicitly specifying the recipient.
  • The set of objects that can handle a request should be dynamically specified.
structure participants
  • Handler
    • Define an interface for handling requests.
    • Implement the successor chain.
  • ConcreteHandler
    • Process requests for which it is responsible.
    • Its successors are accessible.
    • If the request can be processed, process it; otherwise, forward the request to its successor.
  • Client
    • Submit a request to the ConcreteHandler object on the chain.
Effect
  • Reduce coupling.
  • Increased flexibility in assigning responsibilities to objects.
  • Acceptance is not guaranteed.

Example

Use the portfolio model to structure the company based on the diagram below.That is, the head office has a Shanghai East China Branch, and the East China Branch has a Nanjing Office and a Hangzhou Office. Among them, branches at all levels have human resources departments and financial departments. After using the combination model to build the structure, use the chain of responsibility model to handle salary increase requests from employees in various locations. For example, if an employee in Nanjing puts forward a request for a salary increase, if the salary increase does not exceed 1,000 yuan, the Nanjing office can approve it; if it exceeds 1,000 yuan, it will be reported to the East China Branch; if it is within 2,000 yuan, it will be approved; if it exceeds 2,000 yuan, it will be reported to the Beijing head office. If it does not exceed 3,000 yuan, the head office will approve it, otherwise it will be rejected. UMLThe combination mode Component is an object declaration interface, which corresponds to the Handler of the chain of responsibility mode. The code of the two modes can be merged. In addition, by adding two variables, low and bound, to control the number of if statements and reduce the number of classes. Therefore, multiple ConcreteHandlers in the responsibility chain correspond to a specific ConcreteCompany.code 
#include<iostream>
#include<list>
#include <typeinfo>

using namespace std;

class Component{
public:
    Component(string name){
        this->name = name;
    }
    virtual void Add(Component* c) = 0;
    virtual void Remove(Component* c) = 0;
    virtual void Display(int depth) = 0;
    void SetSuccessor(Component* s){
        this->successor = s;
    }
    void SetBound(int low, int high){
        this->low = low;
        this->high = high;
    }
    virtual void HandleReauest(int request) = 0;
protected:
    string name;
    int low, high;
    Component* successor;
};

class Subcom : public Component{
public:
    Subcom(string name) : Component(name){}
    void Add(Component* c){
        cout<<"Cannot add to a leaf";
    }
    void Remove(Component* c){
        cout<<"Cannot remove from a leaf";
    }
    void Display(int depth){
        for(int i = 0; i <= depth; i++)
            cout<<"--";
        cout<<name<<endl;
    }
    void HandleReauest(int request){
        
    }
};

class ConcreteCompany: public Component{
private:
    list<Component*> children;
public:
    ConcreteCompany(string name): Component(name){}
    void Add(Component *c){
        children.push_back(c);
    }
    void Remove(Component *c){
        children.remove(c);
    }
    void Display(int depth){
        for(int i = 0; i <= depth; i++)
            cout<<"--";
        cout<<name<<endl;
        for(Component* child : children){
            child->Display(depth + 1);
        }
    }
    void HandleReauest(int request){
        if(request >= low && request <= high){
            cout<<this->name<<"处理请求: "<<request<<endl;
        }else if(successor != NULL){
            cout<<this->name<<"处理失败"<<endl;
            successor->HandleReauest(request);
        }else{
            cout<<this->name<<"加薪拒绝"<<request<<endl;
        }
    }
};

/* 
即总公司下设上海华东分公司，华东分公司下设南京办事处和杭州办事处。其中，各级分公司均设有人力资源部和财务部。

用组合模式构建好该结构后，利用职责链模式处理各地员工加薪请求。

例如，一南京员工提出加薪请求，如果加薪不超过1000元，南京办事处即可批准；如果超过1000元，则上报华东分公司，如果2000元以内，则批准；超过则上报北京总公司，如果不超过3000元，则总公司批准，否则拒绝。、

实验报告内容:	

 */


int main(){
    Component *root = new ConcreteCompany("总公司");
    root->Add(new Subcom("人力资源部"));
    root->Add(new Subcom("财务部"));
    root->SetBound(2000, 3000);

    Component *huadong = new ConcreteCompany("上海华东分公司");
    huadong->Add(new Subcom("人力资源部"));
    huadong->Add(new Subcom("财务部"));
    huadong->SetBound(1000, 2000);


    Component *nanjing = new ConcreteCompany("南京办事处");
    nanjing->Add(new Subcom("人力资源部"));
    nanjing->Add(new Subcom("财务部"));
    nanjing->SetBound(0, 1000);

    Component *hanzhou = new ConcreteCompany("杭州办事处");
    hanzhou->Add(new Subcom("人力资源部"));
    hanzhou->Add(new Subcom("财务部"));
    hanzhou->SetBound(0, 1000);

    nanjing->SetSuccessor(huadong);
    hanzhou->SetSuccessor(huadong);
    huadong->SetSuccessor(root);
    root->SetSuccessor(NULL);
    huadong->Add(nanjing);
    huadong->Add(hanzhou);

    root->Add(huadong);
    root->Display(1);
    cout<<"*********************"<<endl;
    nanjing->HandleReauest(100);
    cout<<"*********************"<<endl;
    hanzhou->HandleReauest(1500);
    cout<<"*********************"<<endl;
    hanzhou->HandleReauest(2100);
    cout<<"*********************"<<endl;
    hanzhou->HandleReauest(3100);
    return 0;
}

result Summary①This time it is necessary to merge the chain of responsibility mode and the combination mode. The combination mode Component is an object declaration interface and corresponds to the Handler of the chain of responsibility mode. ② The nodes in the combination mode need to be placed in the list of the current parent node. For the container list, the corresponding STL statement needs to be mastered. ③For the chain of responsibility model, in order to reduce the number of classes and coupling, it is necessary to set two upper and lower bounds, low and high. Although there are fewer classes, there are no fewer objects. These objects form a chain, and the request is passed along this chain until an object handles it. Indicates client selection by the name of the current object.
