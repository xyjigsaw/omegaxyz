Easter eggs in Python (hahaha)

Programming LanguagesTechnologyIdeasPythonDeveloper LifePython
Guido van Rossum, the father of Python, invented python in 1989. He liked Monty Python, a six-member British comedy group, the Beatles in the comedy world. Later, the Python interpreter was named IDLE after one of its members. Later, the guys who developed python added easter eggs to the python source code to entertain the public. Let’s take a closer look at the Easter eggs in Python. (Who said programmers have low emotional intelligence!!!)

①Hello World (Hello World)

  >>> import __hello__Output: Hello World! The origin of hello world dates back to 1972, when Brian Kernighan, a famous researcher at Bell Labs, first used it when writing "Tutorial Introduction to the Language B". Python introduced hello world into a built-in module for the first time.

②The Zen of Python

  >>> import thisOutput: The Zen of Python, by Tim Peters Beautiful is better than ugly. Explicit is better than implicit. Simple is better than complex. Complex is better than complicated. Flat is better than nested. Sparse is better than dense. Readability counts. Special cases aren't special enough to break the rules. Although practicality beats purity. Errors should never pass silently. Unless explicitly silenced. In the face of ambiguity, refuse the temptation to guess. There should be one-- and preferably only one --obvious way to do it. Although that way may not be obvious at first unless you're Dutch. Now is better than never. Although never is often better than *right* now. If the implementation is hard to explain, it's a bad idea. If the implementation is easy to explain, it may be a good idea. Namespaces are one honking great idea -- let's do more of those! Translation: Beauty is better than ugliness. Explicit is better than implicit. Simple is better than complex. It's better to be complex than to understand. Flat is better than nested. Sparse is better than dense. Readability should be taken seriously. Although practicality will trump purity, exceptions cannot override the rules. Don't ignore any errors unless you confirm that you want to do so In the face of unclear definitions, resist the temptation to guess. Find the best and only way to solve the problem. Although this approach may not be obvious at first because you are not the father of Python. Doing it well or not doing it, but doing it without thinking is worse than not doing it at all If the implementation is hard to explain, it's a bad idea. If the implementation is easy to explain, it's probably a good idea. Namespaces are a great idea, please take advantage of them. "The Zen of Python" is a book in which core developer Tim Peters condensed Python's language design principles into 20 maxims. It was officially introduced into Python in PEP20, but the actual content is only 19.

④Benefits of C++ programmers

Braces are used to represent code blocks in C++ and Java programming, but the Python community has made a joke for C++ programmers. Braces translates to "curly brackets". You can use braces by importing braces. However, the true attitude of the Python community is: No way!>>> from __future__ import braces SyntaxError: not a chance

⑤April Fools’ Day Easter Eggs

>>> 1<>2 File "<stdin>", line 1 1<>2 ^ SyntaxError: invalid syntax >>> from __future__ import barry_as_FLUFL >>> 1<>2 True >>> 1!=2 File "<stdin>", line 1 1!=2 ^ SyntaxError: invalid syntaxThis is an easter egg in Python3. How to explain it? barry refers to Barry Warsaw, who is also a famous Python developer. FLUFL is the nickname of Uncle Barry. The nickname of the father of Python is BDFL. Uncle Barry believes that the two characters of inequality != are at both ends of the keyboard. Because the two keys are too far apart, it causes finger pain when typing on the keyboard, so Barry hopes to restore the <> symbol in Python2 to represent "not equal to". Therefore, importing barry_as_FLUFL allows you to use <>, but != does not work. For more information, please scan the WeChat official account at the bottom of the page~
