PEP8 Python coding specification

高级语言技术域PythonPython
Why do programmers who habitually press the space bar earn more than programmers who press the Tap key? Because of code specifications! ! ! The following introduces PEP8

one Code layout

1 indent. 4-space indentation (all editors can complete this function), do not use Tap, and cannot mix Tap and spaces. 2 The maximum length of each line is 79. You can use backslashes for line breaks. It is best to use parentheses. The line break point must be hit after the operator. 3. There are two blank lines between class and top-level function definitions; one blank line between method definitions in classes; one blank line between logically irrelevant paragraphs within functions; try to avoid blank lines elsewhere.

two Document layout

1 The order of module content: module description and docstring—import—globals&constants—other definitions. The import part is arranged in the order of standard, third party and self-written, with a blank line in between. 2. Do not use multiple libraries in one import sentence. For example, import os and sys are not recommended. 3. If you use from XX import XX to reference the library, you can omit 'module.', but naming conflicts may occur. In this case, import XX must be used.

three Use of spaces

As a general rule, avoid unnecessary spaces. 1 Do not add spaces before various closing brackets. 2 Do not add spaces before commas, colons, and semicolons. 3. Do not add spaces before the left bracket of the function. Such as Func(1). 4. Do not add spaces before the opening bracket of the sequence. Such as list[2]. 5 Add a space to the left and right of the operator. Do not add spaces for alignment. 6 Omit spaces around the assignment characters used by the default parameters of the function. 7 Do not write multiple statements on the same line, although using ';' allows it. 8 In the if/for/while statement, even if there is only one execution statement, it must start on a new line.

Four Comment

As a general principle, wrong comments are worse than no comments at all. So when a piece of code changes, the first thing to do is to modify the comments! Comments must be in English, preferably complete sentences, with the first letter capitalized. There must be a terminator after the sentence, and the terminator should be followed by two spaces to start the next sentence. If it is a phrase, the terminator can be omitted. 1 block comment, a comment added before a piece of code. Add a space after '#'. Separate paragraphs with lines of only '#'. for example:
# Description : Module config.
# 
# Input : None
#
# Output : None
2 lines of comments, add comments after a line of code. For example: x = x + 1 # Increment x But use this method as little as possible. 3. Avoid unnecessary comments.

five Document description

1. Write docstrings for all public modules, functions, classes, and methods; it is not necessary for non-public ones, but you can write comments (on the next line of def). 2 If the docstring needs to be newline, refer to the following example. For details, see PEP 257
"""Return a foobang

Optional plotz says to frobnicate the bizbaz first.

"""

six Naming convention

As a general principle, new code must be compiled according to the following naming style, and the coding of existing libraries must maintain the style as much as possible. 1 Try to use the lowercase letter ‘l’, the uppercase letter ‘O’ and other easily confused letters alone. 2. Module names should be as short as possible, using all lowercase letters, and underscores can be used. 3. The package name should be as short as possible, using all lowercase letters, and underscores are not allowed. 4. Class naming uses CapWords, and classes used within the module use _CapWords. 5. Exception naming uses CapWords+Error suffix. 6. Global variables should only be valid within the module as much as possible, similar to static in C language. There are two implementation methods, one is the __all__ mechanism; the other is prefixing an underscore. 7 Use all lowercase letters for function names, and underscores can be used. 8 Constants should be named in all uppercase letters, and underscores can be used. 9. Class attributes (methods and variables) should be named in all lowercase letters, and underscores can be used. 9 Class attributes have three scopes: public, non-public and subclass API, which can be understood as public, private and protected in C++. Before non-public attributes, there is an underscore prefix. 11. If the attributes of a class conflict with keyword names, suffix them with an underscore and try not to use abbreviations or other methods. 12 To avoid naming conflicts with subclass attributes, prefix some attributes of the class with two underscores. For example: __a is declared in class Foo, and when accessed, it can only be accessed through Foo._Foo__a to avoid ambiguity. If the subclass is also called Foo, there's nothing you can do. 13 The first parameter of a class method must be self, and the first parameter of a static method must be cls.

seven Coding suggestions

1. Issues such as the efficiency of other Python implementations are taken into consideration during coding. For example, the operator ‘+’ is very efficient in CPython (Python), but very low in Jython, so the .join() method should be used. 2 Use ‘is’ and ‘is not’ instead of ‘==’ whenever possible. For example, if x is not None is better than if x. 3 Using class-based exceptions, each module or package has its own exception class, which inherits from Exception. 4. Do not use naked except in exceptions. Except is followed by specific exceptions. 5. The try code in the exception should be as small as possible. for example:
try:
value = collection[key]
except KeyError:
return key_not_found(key)
else:
return handle_value(value)
Better than:
try:
# Too broad!
return handle_value(collection[key])
except KeyError:
# Will also catch KeyError raised by handle_value()
return key_not_found(key)
6 Use startswith() and endswith() instead of slicing to check sequence prefix or suffix. For example Yes: if foo.startswith(‘bar’): Better than No: if foo[:3] == ‘bar’: 7 Use isinstance() to compare the types of objects. For example Yes: if isinstance(obj, int): Better than No: if type(obj) is type(1): 8 To determine whether a sequence is empty or not, there are the following rules Yes: if not seq: if seq: better than No: if len(seq) if not len(seq) 9. Do not end the string with a space. 10 Use the if boolvalue method to judge binary data. PEP8 official website explains:https://www.python.org/dev/peps/pep-0008/
