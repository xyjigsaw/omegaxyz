A brief description of the differences between Python 2.7.x and 3.x versions

高级语言技术域PythonPython
Most Python libraries support both Python 2.7.x and 3.x versions. Next, I searched some differences between different versions of Python from the Internet.

①__future__ module

Python 3.x introduced some keywords and features that are incompatible with Python 2. In Python 2, these new things can be imported through the built-in __future__ module. If you want code written in a Python 2 environment to also run in Python 3.x, it is recommended to use the __future__ module. For example, if you want to have the integer division behavior of Python 3.x in Python 2, you can import the corresponding module with the following statement.
from __future__ import division
(source:https://docs.python.org/2/library/future.html)

②print function

Although the print syntax is a small change in Python 3 and should be well known by now, it's still worth mentioning: the print statement in Python 2 was replaced by the print() function in Python 3, which means that in Python 3 the object to be output must be enclosed in parentheses. Using extra parentheses is also possible in Python 2. But conversely, when you want to call the print function in Python 2 without parentheses in Python 3, a SyntaxError will be triggered.
Python 2 
print 'Python', python_version()
print 'Hello, World!'
print('Hello, World!')
print "text", ; print 'print more text on the same line'
#结果---------------
Python 2.7.6
Hello, World!
Hello, World!
text print more text on the same line
Python 3 
print('Python', python_version())
print('Hello, World!')
  
print("some text,", end="") 
print(' print more text on the same line')

#结果-------------
Python 3.4.1
Hello, World!
some text, print more text on the same line
Notice:In Python, it is normal to output "Hello World" with or without parentheses. But if you print multiple objects at the same time in parentheses, a tuple will be created. This is because in Python 2, print is a statement, not a function call.

③Integer division

Since people often overlook the changes in integer division in Python 3 (a Syntax Error will not be triggered if written incorrectly), you need to pay special attention to this change when porting code or executing Python 3 code in Python 2. Therefore, I will still try to use float(3)/2 or 3/2.0 instead of 3/2 in the Python 3 script to avoid errors that the code may cause in the Python 2 environment (or conversely, use from __future__ import division in the Python 2 script to use Python 3 division).

④xrange

In Python 2.x, xrange() is often used to create an iterable object, usually in a "for loop" or "list/set/dictionary comprehension". This behavior is very similar to generators (like "lazy evaluation"), but here the xrange-iterable is endless, meaning that it is possible to iterate infinitely over this xrange. Due to the "lazy seeking" feature of xrange, range() is usually faster than xrange() if it only needs to be iterated once (such as in a for loop). However, it is not recommended to use range() over multiple iterations, because range() will regenerate a list in memory each time. In Python 3, range() is implemented in the same way as the xrange() function, so there is no dedicated xrange() (using xrange() in Python 3 will trigger a NameError).

⑤For loop variables and global namespace leaks

The good news is: in Python 3.x, variables in for loops no longer leak into the global namespace! This is a change made in Python 3.x, described as follows in "What's New In Python 3.0": "List comprehensions no longer support the syntax [... for var in item1, item2, ...], use [... for var in (item1, item2, ...)] instead. Also note that list comprehensions have different semantics: List comprehensions are now closer to the syntactic sugar of generator expressions in the list() constructor. In particular, note that loop control variables no longer leak into the space around the loop."

⑥Parse user input through input()

Fortunately, Python 3 improved the input() function so that it will always store user input as a str object. In Python 2, in order to avoid some dangerous behaviors that occur when reading non-string types, raw_input() has to be used instead of input().

⑦Return iterable objects instead of lists

As you can see in the xrange section, some functions and methods return iterable objects in Python instead of lists in Python 2. Since these objects are usually traversed only once, this approach saves a lot of memory. However, iterating through these objects multiple times through a generator is inefficient. At this point we really need a list object, and we can simply convert the iterable object into a list through the list() function.
