Comparative analysis of software engineering life cycle models

TechnologySoftware EngineeringSoftware Engineering

Just like anything, software also has its own survival process of gestation, birth, growth, maturity and decline, which is generally called the "software life cycle". The software life cycle is generally divided into six stages, namely planning, requirements analysis, design, coding, testing, operation and maintenance. The relationship between the various stages of software development cannot be sequential and linear, but should be an iterative process with feedback. In software engineering, this complex process is described and represented using software development models.

Iterative-incremental life cycle model

Strengths:①The model closest to real-world software development, containing a unified process approach ② Provide multiple opportunities to check the software product, and each iteration includes a test flow ③The robustness of the underlying structure can be determined at an early stage ④ Able to mitigate risks earlier ⑤We always have a working version of the softwareKeywords:Miller's rule, step-by-step refinement

incremental model

For a limited time, a set of versions with limited functionality is constantly being expanded.Conditions of use:Limited development time, insufficient developers, and a clear initial modelTask flow:Linear, parallel (requirements are continuously added to the task flow)Note: ①The first increment is the core product, which meets the basic needs of users. ②Evaluate and formulate incremental plans based on customer use. ③Each increment is a runnable product.

evolutionary tree model

Strengths:The closest model to real-world software development, equivalent to the iterative incremental model.

Coding - patching lifecycle model

Strengths:Suitable for small programs that do not require any maintenanceDisadvantages:Overall not suitable for important programsNote:Maintenance is expensive

Waterfall life cycle model

Systematic sequential development methods Conditions of use:① User needs can be clearly understood ② Requirements must be accurately defined and relatively stableTask flow:linear task flowNote:①Continuous changes in inapplicable items ②Not applicable to projects with uncertain demand ③Do not violate the development sequence ④Document-driven (also a weakness)Strengths:Disciplined and enforced approach, document-driven (also a weakness)Disadvantages:Products delivered may not meet customer requirements

Rapid prototyping life cycle model

Each iteration produces a more complete versionConditions of use:① Users put forward the most basic requirements, and no more specific functional characteristics are defined. ② The operating system is not sure about the efficiency of the algorithm.Task flow:iteration/(linear)Note:① Developers build prototypes at will ② In order to speed up the construction, use inappropriate OS ③ Speed ​​is the keyStrengths:Ensure delivered products meet customer requirementsDisadvantages:Not yet proven invulnerable

Open source life cycle model

Keywords:①Reporting and correcting shortcomings is corrective maintenance ②Adding additional functions is complete maintenance ③ Converting the interface to a new environment for the program is adaptive maintenanceStrengths:Works reasonably well on a small number of instances (Linux, Apache, gcc)Disadvantages:Limited utility, usually doesn't work

Synchronous-stable life cycle model

Keywords:sync, stabilize, freezeStrengths:Meet future user requirements and ensure successful integration of componentsDisadvantages:Not widely used except at Microsoft

agile process

Respond effectively to change + philosophical thinkingkey:① Effective communication ② Emphasis on rapid delivery of final products ③ Integrate customers into the team ④ Flexible adjustment of plansFeatures:①Small team - small but highly autonomous ②Fast product – a system that can be delivered quickly and successfully ③ Process spirit - minimize software work products, streamline overall development, and respond better to changes.Keywords:①Test-driven development (TDD): Develop test cases for tasks ②Pair programming ③Timeboxing: Frequent software delivery ④Stand-up meetingStrengths:Works well when customer needs are vagueDisadvantages:Seems only suitable for small projects

spiral life cycle model

Strengths:risk drivenDisadvantages:Only suitable for large-scale internal software products, developers must be proficient in risk analysis and risk eliminationThe radial coordinate represents the accumulated cost so far, and the angular coordinate represents the progress of the spiral. Each turn of the spiral corresponds to a stage. Each stage starts with (top left) determining the goals, implementation methods, and constraints of the stage. If the risk is successfully reduced, it will enter the next stage (bottom right)
